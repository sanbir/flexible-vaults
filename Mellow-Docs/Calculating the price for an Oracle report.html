Calculating the price for an Oracle report

This document describes how we compute the price that an off-chain service will report to the on-chain Oracle. Throughout the document asset denotes the vault’s underlying ERC-20 token (e.g. USDC, wETH etc.) and shares represents LP tokens issued by the vault.
A single vault can manage multiple ERC-20 tokens (e.g. wstETH, wETH, USDC) at the same time. Every token has its own set of deposit / redeem queues and therefore its own price report. The off-chain reporter must compute the formula (see
High-level formula
section below) independently for each
asset
and submit the result to the oracle via
Oracle.submitReports(Report[])
.

Relation of
shares
and
assets

The reported price is structured such that the fundamental invariant holds for each token:
shares = (assets * priceD18) / 1e18
​
Consequently, the price we quote to the Oracle is simply the reciprocal of the exchange rate
shares / assets
. When the vault earns yield the denominator grows and the ratio falls – signaling that each share is now backed by more underlying liquidity.
For a vault, the natural notion of “price” is how many shares correspond to one unit of asset. As the vault earns yield, the number of assets grows while the share supply stays the same and the ratio therefore decreases. A falling
priceD18
therefore means each share is backed by more assets, which is exactly what we want depositors to observe.

High-level formula

priceD18 = (totalShares + incomingFees(baseAsset, priceD18)) * 1e18 / (totalAssets(asset) − totalRedeemDemand(asset))
​
priceD18
is the value reported to the Oracle and is expressed with 18 share decimals.
Because each asset can have a different number of decimals (e.g. 6 for USDC, 18 for DAI),
priceD18
scales to compensate for this mismatch. As a result, for assets with fewer decimals the reported value can be much larger than
1e18
.
For instance, suppose the vault has
1e18
shares representing
1e18
wei of DAI (both tokens use 18 decimals). A user now deposits
1e6
wei of USDC (which equals 1 USDC because USDC has 6 decimals). To mint the correct amount of shares for USDC, we need to report the
priceD18 = 1e30
.
The rest of this document explains every term in the formula and the design choices behind it.

1.
totalShares

This is the current on-chain supply of shares (minted vault’s LP tokens):
totalShares = vault.shareManager().totalShares()
​

2.
totalAssets(asset)

The total assets in the formula are technically an estimate of Total Value Locked (TVL) denominated in that specific token.
Its value aggregates:
Idle liquidity held by the root vault.
Idle liquidity held by all subvaults.
Open positions on external integrations (Uniswap, Symbiotic, Gearbox, Aave, …) converted to asset terms.

TVL denomination example

Imagine a vault with such balances:
1000 wei of token0 (e.g. wstETH)
1050 wei of token1 (e.g. wETH)
Market price assumption: 1 wei of token0 is worth 1.18 wei of token1, then:
totalAssets(token0) = 1000 + (1050 / 1.18) = 1889.83 totalAssets(token1) = 1050 + (1000 * 1.18) = 2230
​
The total value of assets denominated in token0 is lower because the price of token0 is higher compared to token1.

3.
totalRedeemDemand(asset)

This is the total value of all redemption requests — in all withdrawal queues — that have been created but not yet processed by
RedeemQueue.handleBatches(n)
. The vault is economically obliged to pay out these assets, so they can no longer be considered part of the vault’s backing:
totalRedeemDemand(asset)
should be calculated in the same way as
totalAssets(asset)
, i.e. denominated in the specific asset.

4.
incomingFees(baseAsset, priceD18)

NOTE: Using only when price is calculated for the base asset. For any other asset this value should be 0.
Protocol and performance fees are charged in shares when the oracle report is handled (
ShareModule.handleReport
), i.e. after the price is calculated. To maintain the correct ratio between shares and assets, we must take these fees into account in advance.
Basically, the
incomingFees
is the value we can get by calling:
FeeManager.calculateFee(vault, asset, priceD18, totalShares)
​
vault
– The address of the vault for which we are calculating prices.
asset
– The address of the asset on which we calculate, i.e. the vault's base asset.
priceD18
– Currently unknown value, the price that we want to calculate, see
Solving the recursive formula
section for details.
totalShares
– The total number of LP tokens minted by the vault, minus the fee recipient's balance (we should subtract the recipient's active shares to achieve smoother calculations and avoid the compound effect of the fees):
totalShares = shareManager.totalShares() - shareManager().activeSharesOf(feeManager.feeRecipient())
​

Solving the recursive formula

Because
incomingFees
itself depends on the yet unknown
priceD18
, the equation is recursive. Three practical ways to obtain a solution are:
Iterative approximation – Start with the previously reported price as an initial guess and iterate until the value converges (e.g. if the change is below 1 wei).
Binary search – A bounded binary search applies. This is valid because the fee‑adjusted price is monotonic non‑increasing: as the candidate price rises, performance fees shrink and reach zero at the minimum‑price threshold, while protocol fees remain constant. The minimum price gives a natural upper bound. If that value already satisfies the formula, use it; otherwise search within the bounds for the smallest price that satisfies the formula. We chose this approach when developing our price calculation utility (see the
Oracle helper
section below).
Closed-form solution – An analytical solution exists, but the above-mentioned methods are simpler, more reliable in production, and not heavily tied to the details of the
calculateFee
implementation.

Calculating prices for vaults with multiple assets

When a vault holds several different assets, a more convenient way to calculate the prices for the oracle is to pick one “base asset” and express all prices relative to it, i.e.:
Establish the base asset’s price using the
High-level formula
mentioned above.
Derive prices for other assets. With the base price set, convert every other asset using its safe market rate to the base. If 1 unit of Asset X equals R units of the base asset, then Asset X’s price is simply the base price scaled by R.

Oracle helper

There is the OracleHelper.sol contract that provides the utility
getPricesD18
function, which does exactly what is mentioned above – it computes oracle prices for all vault assets anchored to a single base asset.
OracleHelper.getPricesD18( Vault vault, uint256 totalAssets, AssetPrice[] calldata assetPrices ) → uint256[] pricesD18
​
vault
: the target Vault.
totalAssets
: TVL denominated in the base asset units. For multiple assets, first convert each asset’s amount to the base using its market rate and sum them. Example: if 1 unit of X equals R base units, add amountX × R to 
totalAssets
.
assetPrices
: sorted list of (asset, priceD18):
Exactly one entry must have 
priceD18
=  0 (the base asset). Use
feeManager.baseAsset(...)
if it’s set.
For any non-base asset X, 
priceD18
 is its market rate to the base in 18-decimals.
If
priceD18
is 1e18 – 1 unit of X = 1 unit of base.
If
priceD18
is 0.5e18 – 1 unit of X = 1/2 unit of base.
If
priceD18
is 2e18 – 1 unit of X = 2 unit of base.
This function returns an array
uint256[] pricesD18
of the prices that should be passed to the oracle.
Note the constraints:
assetPrices
 must be strictly sorted by asset ascending.
Exactly one base asset should be specified in
assetPrices
.
Every asset used by the vault’s queues must appear in 
assetPrices
.
