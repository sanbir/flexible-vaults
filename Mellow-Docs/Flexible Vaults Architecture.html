Flexible Vaults Architecture

Overview

Flexible Vaults are built on a modular, upgradeable architecture that enables seamless integration with new protocols. The system is designed to offer dynamic permission management for curators, allowing precise and adaptable control over what actions can be performed — without compromising on safety or decentralization goals.
This architecture implements a fully-featured liquid vault with the following capabilities:
Granular Role-Based Access Control – Vault behavior is governed by configurable roles and permissions that define what users and curators are allowed to do
Deposits – Users can convert supported assets into vault shares via a request queue mechanism
Withdrawals – Shares can be redeemed back into underlying assets, subject to queue processing and oracle pricing
Delegated Yield Strategies – Curators can interact with external protocols to generate yield on the liquidity in the vault

Granular Role-Based Access Control

Flexible Vaults introduce
Verifier.sol
, a call-level permission control system. It determines whether a curator (CALLER_ROLE holder) is allowed to make specific calls based on the verification type:

Verification Types

ONCHAIN_COMPACT
Allows calls where
hash(who, where, selector)
exists in an admin-controlled allowlist.
Primarily for static strategies or system-owned contracts.
MERKLE_COMPACT
Like ONCHAIN_COMPACT, but uses Merkle root validation for scalability.
Allows large call sets with minimal gas overhead.
MERKLE_EXTENDED
Validates
hash(who, where, value, calldata)
against a Merkle tree.
Ensures full calldata and value are verified.
CUSTOM_VERIFIER
Delegates permission checks to contracts implementing
ICustomVerifier
.
Enables arbitrary verification logic.

Built-in Custom Verifiers

: Validates

Oracle

Flexible Vaults rely on a permissioned oracle for pricing and share conversions. The oracle:
Accepts periodic reports from a trusted operator
Validates reports against absolute/relative deviation thresholds:
If out of
maxDeviation
→ reverts
If out of
suspiciousDeviation
→ requires explicit admin acceptance
It is expected that oracle works only with correlated assets (e.g., ETH/wstETH/stETH) to avoid manipulation
Triggers processing of deposit/redeem queues on report acceptance
See the “Calculating the price for an Oracle report” page for how input prices are computed prior to being reported on-chain to the oracle.

Deposit Queue

Handles user deposits through a time-buffered queueing system.
Each user submits a deposit request defined by
(amount, timestamp)
Only one active request per asset is allowed per user
A request is processed by the first oracle report whose timestamp is at least
depositInterval
seconds after the deposit’s timestamp
This delay protects against front-running and price manipulation by ensuring deposits aren't executed using stale or potentially influenced price data
Fenwick Tree is used to:
Track prefix sums of deposits across timestamps
Aggregate total deposits eligible for a specific oracle report
Flow:
User deposits tokens — the request is stored and marked as pending.
The next valid oracle report triggers processing of pending requests that were submitted at least
depositInterval
seconds earlier.
Corresponding shares are allocated (not yet minted) to the user.
The user (or anyone on their behalf) can call
claim(account)
to mint their allocated shares.
Early cancellation is possible — until a request is processed, the user may call
cancelDepositRequest()
to pull back the assets and delete the request.

Redeem Queue

Handles withdrawal requests.
Users submit a redeem request:
(shares, timestamp)
Requests cannot be cancelled (prevents yield-griefing)
Multiple concurrent requests allowed
Flow:
User submits redeem request
Oracle report arrives with valid price
Curator pulls liquidity from protocols
handleReports()
allocates assets to the queue
Users call
claim(receiver, timestamps)
to withdraw assets
Prefix sums (array-based) used to track redemption requests since cancellations aren't allowed.

Fees

The system defines four fee types, each triggered under specific conditions during the vault lifecycle:
protocolFee
– Applied to all staked shares when a new oracle report is processed. This fee accrues linearly based on a fixed annualized rate
performanceFee
– Triggered when a new oracle report pushes the share price above the previous high-water mark (HWM), and applied to the appreciated value
depositFee
– Collected on all deposits processed through the
DepositQueue
, deducted from the deposited amount
redeemFee
– Collected on all withdrawals processed through the
RedeemQueue
, deducted from the redeemed asset amount
