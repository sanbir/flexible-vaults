How do Core Vaults’ isolated subvaults enforce per-strategy limits (whitelists, allowed call targets, debt ceilings)?
Simple version: Each
Subvault
is an isolated module with its own rules. It can only hold a limited number of shares, accept deposits in a predefined set of assets, and execute a restricted list of calls. These rules are enforced by a
Verifier
contract, which checks whether actions are permitted. The verifier can use simple allowlists of approved calls, Merkle-tree proofs for more complex validation, or delegate checks to an external custom verifier. This ensures that each strategy runs within strict boundaries and cannot interact with unauthorized assets or contracts.
Extended version: Each isolated
Subvault
has:
its own share limit,
a defined set of depositable assets,
a set of allowed calls (addresses, selectors) defined by its
Verifier
contract,
and, optionally, its own dedicated
Verifier
contract.
The
Verifier
supports:
Onchain allowlists using hashed, compacted calls (
CompactCall
contains
who
,
where
, and
selector
),
Merkle tree–based validation for compact Merkle (
who
,
where
,
selector
), extended Merkle (
who
,
where
,
value
,
data
),
and a custom verification type delegating logic to an external verifier (
ICustomVerifier
).

Which oracle providers are supported, and how do the “hard” vs. “suspicious” deviation checks work in code (thresholds, cooldowns)?
There is no fixed list of oracle providers. Any trusted entity (that is set at deployment or by the vault admin later) can act as a provider (submit reports) and an acceptor (validate reports). A distinct role configures security parameters, including suspicious and maximum intervals for absolute and relative deviations.
The flow is:
A submitter submits a report. If the price change is outside the hard range, the transaction reverts.
If the report is valid and not suspicious, it is automatically accepted and processed by the Vault.
If the report is flagged as suspicious, it must be explicitly accepted by an authorized acceptor (that is set at deployment or by the vault admin later) in a separate transaction before being processed.
Additional parameters:
timeout
defines the reporting frequency (typically 24 hours).
depositInterval
defines the minimum age (in seconds) a deposit request must reach before it can be processed with a submitted report.
redeemInterval
defines the equivalent minimum age for redemption requests.
How are batch redemptions priced to avoid last-look or sandwich risk, and what are the settlement cadences?
Redemption requests are queued and processed in batches at the next oracle update. Only requests older than the minimum interval (
redeemInterval
) are included, and all requests in the batch receive the same price. This mechanism prevents last-look or sandwich attacks. Liquidity settlement occurs in a separate step: it may be immediate after the price report or staged (e.g., daily or as positions unwind). The batch price is fixed at the oracle report. Settlement cadence follows the oracle/report schedule, typically configurable between 1 day and 2 weeks.
Who qualifies for the fast-lane deposit/withdraw path, what rate limits apply, and how is this surfaced to LPs to avoid perceived unfairness?
Fast-lane access is limited by consensus multisig set that approves any order. The
Consensus
contract, owned by the vault proxy admin, defines the set of signers and quorum requirements.
Each order is further validated by the
Oracle
. Orders are only accepted if their price falls within the non-suspicious range (absolute and relative), ensuring fairness relative to the last accepted report.
For strategies that touch centralized venues, how does custody segregation, rehypothecation restriction, etc. work?
Execution on centralized exchanges occurs via Copper ClearLoop or equivalent infrastructure. Custody is secured by a three-party Multi-Party Computation (MPC) scheme, with key shares distributed among the exchange (CEX), Copper, and the strategy manager (Mellow/Curator). Under MPC, no single party ever controls the private key; transactions require joint cryptographic authorization. This enforces custody segregation, ensuring client assets remain segregated and not commingled. It also prevents rehypothecation, as neither the CEX nor any single participant can deploy assets unilaterally. The model enhances operational resilience and mitigates counterparty and misuse risks in centralized venue interactions.
